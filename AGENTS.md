# BigV Agent 开发规范

## Git 提交

- 提交标题用中文短句概括结果。
- 提交正文按列表简短写具体改动；无必要不写正文。
- 提交前按改动范围选择是否检查 diff；涉及多人协作、合并冲突或跨模块改动时，确认 diff 只包含本次改动。
- 提交前按任务复杂度选择是否做代码自检；涉及逻辑调整、重构或公共模块时，检查是否存在可删除的冗余逻辑、重复分支、无效辅助函数或代码复述型注释。

## 产品边界

- BigV 的 V1 目标是本地单机、单直播间、单操作者，不提前设计多租户、云调度和多直播间并发。
- Provider 边界固定为 OpenAI 兼容 HTTP 接口，不提前引入厂商私有协议。
- 主播语音是主输入链路；Echo-Live 输入和视觉上下文是增强源，不得在实现中反转优先级。
- 互动输出类型固定为 `danmaku`、`gift`、`super_chat`、`membership` 4 类。
- 记忆系统固定分为主播设定、长期互动事实、观众画像、场次摘要 4 层。

## 代码

- 先读相关模块、数据契约、现有测试和对应文档，再动手改代码。
- 先确认跨端数据契约，再同步前端、后端和测试。
- 不做打补丁式修复；先定位问题在哪一层，再在正确边界修正。
- 不把“看起来像直播”当成完成，必须区分生成、节奏控制、投递渲染是否真实落地。
- 改 blivechat 相关前端时，优先保证事件契约和渲染链路清晰，不顺手扩展无关 UI。
- 记忆系统改动必须明确读写时机、摘要策略和污染风险，不能只增加字段而不定义写回规则。
- 新增组件或功能单元时，优先解耦到独立文件或模块；仅在任务明确要求时才在现有大组件内追加逻辑。
- 优先沿用现有结构和命名，不顺手做无关重构。
- 不覆盖用户或其他 Agent 的已有改动。

## 架构与文档

- 架构讨论与实现以 `docs/architecture.md` 为准；如实现偏离文档，必须同步更新文档。
- TODO 拆分与开发优先级以 `docs/todo.md` 为准，不临时扩展到文档未定义的子系统。
- 需要长期记录的背景、取舍、限制和未决问题，优先写进文档，不写成代码注释。
- 对外描述和内部命名尽量统一使用：主播语音、Echo-Live、视觉上下文、观众模拟、长期记忆、互动投递。

## 样式

- 保持工程工具气质：克制、清晰、可扫描。
- 视觉分级明确：直播状态与互动流 > 当前异常与配置 > 调试细节与辅助操作。
- 优先使用现有 CSS 变量、间距和组件语言；深浅主题都要可读。
- 涉及 UI 设计或交互模式调整时，先确认目标交互，再实现。

## 验证

- 功能实现后根据任务风险和影响范围选择验证；可选验证包括定向测试、`yarn test`、`yarn build`、`cargo check --manifest-path src-tauri/Cargo.toml`、`yarn docs:build` 或 `yarn verify`。
- 文档、注释、配置说明等低风险改动可不跑全量验证；涉及持久化、provider 接入、记忆写回、事件投递或构建配置时，优先运行最小必要验证。
- 若未运行测试、构建或验证，在最终说明里写清楚原因；若验证无法运行，写清楚阻塞原因和剩余风险。
