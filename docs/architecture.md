# BigV 架构文档

## 1. 目标

BigV 的 V1 目标是在本地单机环境中，为单个主播和单个直播间提供可控的 AI 观众模拟闭环。

闭环定义为：

1. 接收主播语音文本。
2. 结合 Echo-Live 与视觉摘要形成上下文。
3. 检索长期记忆并组装提示。
4. 生成观众互动意图。
5. 经过节奏与真实性约束后投递到改造后的 blivechat 前端。

V1 不包含多直播间、云端编排、房管活动系统或厂商私有协议适配。

## 2. 固定边界

- 部署形态：本地单进程实现，单机、单直播间、单操作者。
- 主输入：主播语音转文本。
- 增强输入：Echo-Live、视觉上下文摘要。
- 输出类型：`danmaku`、`gift`、`super_chat`、`membership`。
- Provider：OpenAI 兼容 HTTP 接口。
- 记忆层次：主播设定、长期互动事实、观众画像、场次摘要。
- 架构原则：首版本地实现，但接口按可插拔模块定义。

## 3. 系统分层

### 3.1 输入适配层

负责把外部输入统一转成内部上下文事件。

- ASR 文本源：接收主播语音转文本结果，作为主输入。
- Echo-Live 输入源：接收外部辅助文本、状态或控制信号。
- 视觉上下文源：接收对直播画面的识别摘要，而不是原始图像流。

产出统一的输入事件：

- 来源
- 时间戳
- 内容摘要
- 置信度或可用性状态

### 3.2 上下文汇聚层

负责把离散输入事件合成为当前直播时刻可消费的上下文窗口。

维护内容包括：

- 最近主播语音片段
- 最近 Echo-Live 事件
- 最近视觉摘要
- 当前直播状态
- 当前时间窗内已投递互动
- 当前场次的短期摘要

该层不负责生成互动，只负责回答“现在系统认为直播间正在发生什么”。

阶段 3 最小实现：

- Tauri 命令 `submit_context_event` 接收统一上下文事件输入。
- Tauri 命令 `load_context_window` 读取当前运行态时间窗。
- Tauri 命令 `clear_context_window` 清空运行态时间窗，便于本地验证。
- 当前只把 `voice` 主播语音文本写入窗口；`echo_live` 与 `vision` 保留同一事件模型和状态展示，不产生真实上下文事件。
- 时间窗默认保留最近 300 秒、最多 80 条事件；主播语音 60 秒内有输入视为在线，300 秒内仅有旧输入视为空闲，无事件视为待输入。

### 3.3 记忆层

负责长期可持续的信息读写。

固定 4 类记忆：

- 主播设定：主播人设、语言风格、禁忌、直播主题。
- 长期互动事实：历史梗、稳定关系、反复出现的互动偏好。
- 观众画像：观众身份、语气、消费倾向、出现频率。
- 场次摘要：单场直播内的重要事件和收束摘要。

记忆层职责：

- 为生成前检索提供结构化上下文。
- 在每轮互动后决定是否回写。
- 对冲突、低置信度和污染风险做标记。

### 3.4 模拟编排层

负责决定“该不该发”和“该由谁发”，不直接负责自然语言生成。

核心职责：

- 管理观众人设池和同接规模。
- 判断当前直播处于冷场、平稳、高潮或收束状态。
- 控制各类互动频率、概率、间隔和并发。
- 为本轮选择候选观众、互动类型和强度。
- 拒绝不合理的高频输出、冲突行为和过度消费行为。

### 3.5 生成层

负责把编排意图和上下文送入 OpenAI 兼容 provider，并解析结果。

核心职责：

- 组装系统提示、上下文提示和结构化输出约束。
- 调用 provider。
- 解析结构化互动结果。
- 在失败时返回可诊断错误，而不是直接吞掉。

生成层不负责决定系统节奏，只负责在给定意图下生成内容。

### 3.6 投递层

负责把结构化互动事件送入前端渲染链路。

投递职责：

- 统一事件格式。
- 将事件路由到弹幕、礼物、SC、舰长通道。
- 维护前端事件队列和渲染顺序。
- 记录投递成功、失败或节流结果。

投递层以改造后的 blivechat 前端为主要目标，不在 V1 承诺真实平台对接。

阶段 6 最小实现：

- 前端实现本地 `EventEmitter` 与 blivechat 事件队列，不接真实平台。
- 本地队列记录 `enqueue`、`deliver`、`throttle` 三类协议动作。
- 弹幕姬页面的互动时间线与投递统计卡片由本地队列快照驱动，不再读取静态投递快照。
- 默认本地模拟覆盖 `danmaku`、`gift`、`super_chat`、`membership` 4 类互动；SC 与礼物类事件仍受本地开关节流。
- 本地 mock 数据源支持手动启动、暂停、单步和重置，循环触发上下文测试输入与 4 类互动输出；该能力只用于本地验证，不代表真实平台或外部 blivechat 源接入。

## 4. 核心抽象

以下抽象名在后续实现中固定保留，允许语言级实现不同，但职责不能漂移。

### `ContextSource`

输入：外部原始信号，例如 ASR 文本、Echo-Live 事件、视觉摘要。
输出：统一的上下文事件流。

职责：

- 声明来源类型和可用性。
- 提供时间戳和原始内容摘要。
- 将不同来源转换为统一结构。

阶段 3 固定的来源类型为 `voice`、`echo_live`、`vision`。其中 `voice` 是唯一已接入主链路的输入；`echo_live` 和 `vision` 只作为后续增强输入的挂接口存在。

### `MemoryStore`

输入：检索条件、候选写回内容、冲突标记。
输出：结构化记忆片段、写回结果、冲突信息。

职责：

- 分层读写 4 类记忆。
- 支持按直播上下文检索。
- 标记冲突、置信度和污染风险。

### `AudiencePlanner`

输入：当前上下文窗口、长期记忆、最近互动记录、节奏状态。
输出：待执行互动意图列表。

职责：

- 选择候选观众和互动类型。
- 控制频率、间隔和同接节奏。
- 给生成层提供明确意图，而不是泛化请求。

### `ProviderClient`

输入：Provider 配置、已选择模型、提示词、结构化输出要求。
输出：结构化生成结果或错误。

职责：

- 前端通过 Tauri 命令读写 Provider 配置。
- 从远端模型列表获取可选模型。
- 调用 OpenAI 兼容 HTTP 接口。
- 处理认证、模型选择和返回解析。
- 通过 `responseFormat` 承载结构化输出约束；阶段 2 先打通 `text` 与 `jsonObject`。
- 输出统一错误模型，便于调试和回退。

### `EventEmitter`

输入：结构化互动事件。
输出：前端渲染事件、投递状态。

职责：

- 将结果分发到弹幕、礼物、SC、舰长通道。
- 保证事件顺序和可追踪性。
- 记录投递成功、失败、节流或丢弃原因。

当前前端最小实现只承载本地模拟投递：`enqueue` 写入前端队列，`deliver` 形成已投递记录，`throttle` 记录通道关闭或自动投递暂停导致的节流原因。

## 5. 默认数据流

默认单轮流程如下：

1. `ContextSource` 接收主播语音文本，必要时合并 Echo-Live 与视觉摘要。
2. 上下文汇聚层更新当前时间窗与短期场次状态。
3. `MemoryStore` 按当前上下文检索主播设定、长期事实、观众画像和场次摘要。
4. `AudiencePlanner` 结合上下文、最近互动记录和节奏状态，生成候选互动意图。
5. 生成层组装提示词和结构化输出约束。
6. `ProviderClient` 调用 OpenAI 兼容 provider，返回结构化互动结果。
7. 模拟编排层对结果执行频率与真实性约束，必要时拒绝、延后或降级。
8. `EventEmitter` 将最终事件投递到改造后的 blivechat 前端。
9. 记忆层根据结果和阈值决定是否写回长期记忆或场次摘要。

本地测试流程可由 mock 数据源代替第 1、4、5、6、7 步中的外部输入和生成结果：它按固定场景循环提交主播语音、Echo-Live、视觉上下文测试数据，并把预置互动事件送入同一投递队列。Echo-Live 与视觉上下文仍遵守阶段 3 的预留契约，不写入真实上下文窗口。

## 6. 关键数据对象

V1 文档层面固定以下对象，不在本轮细化到字段级 schema，但后续实现不得改变其职责。

### 互动事件

- `danmaku`
- `gift`
- `super_chat`
- `membership`

每个事件至少要有：

- 事件类型
- 观众身份
- 内容或金额
- 触发时间
- 来源轮次或生成依据

### 观众画像

- 身份标签
- 语言风格
- 消费倾向
- 出现频率
- 与主播关系

### 场次状态

- 当前时间窗摘要
- 最近输出密度
- 当前直播节奏
- 最近异常与回退状态

## 7. 写回与污染约束

- 不是每条互动都写回长期记忆。
- 场次摘要优先承接短期事实，长期事实只写入稳定、重复或明确被主播确认的信息。
- 观众画像更新必须可追溯到生成结果或显式规则，不能让模型随意覆写已有稳定画像。
- 冲突事实进入待确认或低置信度状态，不直接覆盖旧值。

## 8. 可观测性要求

V1 至少需要能追踪：

- 当前输入源是否在线。
- 最近一次 provider 调用是否成功，以及阶段 2 连通性测试是否通过。
- 当前节奏状态和拒绝原因。
- 最近一次投递结果。
- 记忆检索与写回是否成功。

调试面板的目标是帮助回答“为什么这条互动会出现或没出现”。

## 9. 实现约束

- 先保证主播语音主链路闭环，再接入增强输入。
- 先保证单轮生成正确，再扩展复杂节奏策略。
- 先保证投递链路稳定，再做额外样式和装饰。
- 任何实现若偏离本文档固定边界，必须同步更新文档和 TODO。
