# BigV 开发 TODO

本文档只记录 V1 主闭环的可执行路线，不记录远期愿景。

## 阶段 1：基础重命名与壳层改造

目标：把当前初始桌面壳转换为 BigV 的产品语义和导航骨架。

- 更新应用名、产品标题、图标、配置前缀和文案。
- 重命名首页、设置页、扩展页等初始占位语义。
- 规划 BigV 的基础导航：弹幕姬、额度检查、观众信息、直播回顾、设置。
- 保留现有 Tauri/Vue 壳结构，不提前引入业务逻辑耦合。

完成判定：

- 桌面壳与文档不再出现初始工程命名。
- 基础导航能映射到 BigV 的真实页面职责，并不再保留模板占位页面。

## 阶段 2：Provider 与配置

目标：建立 OpenAI 兼容 provider 的最小可用接入。

- 定义 provider 配置项：Base URL、API Key、远端模型选择。
- 实现连通性测试和错误回显。
- 固定 `ProviderClient` 的统一输入输出与错误模型。
- 明确结构化输出约束的承载方式。
- 模型名必须从 OpenAI 兼容模型列表获取后选择。

完成判定：

- 能获取远端模型列表，选择模型，并发起一次 OpenAI 兼容请求看到明确结果或错误。
- 上层模块无需关心具体厂商差异。

## 阶段 3：输入接入

目标：打通主播语音主链路，并预留增强输入入口。

- [x] 接入主播语音文本输入，形成统一上下文事件。
- [x] 接入 Echo-Live WebSocket 文本输入，作为增强上下文事件。
- [x] 预留视觉摘要输入接口，不直接处理原始视频流。
- [x] 建立上下文时间窗和最近事件聚合逻辑。

阶段 3 已落地范围：

- 本地 Tauri 命令 `submit_context_event`、`load_context_window`、`clear_context_window`。
- 弹幕姬页面的主播语音文本手动接入口。
- 最近 300 秒、最多 80 条的运行态上下文窗口。
- 输入源状态展示：主播语音和 Echo-Live 随窗口变化，视觉上下文保持预留接口状态。
- Echo-Live 通过本机 `ws://127.0.0.1:3000` 手动连接，消费 `message_data` 文本。

完成判定：

- 系统能稳定消费主播语音文本并更新当前上下文窗口。
- Echo-Live 输入可按同一事件模型进入上下文窗口，视觉输入保留同一挂接模型。

## 阶段 4：记忆系统

目标：建立结构化长期记忆和场次摘要能力。

- [x] 定义 4 类记忆的存储边界：主播设定、长期互动事实、观众画像、场次摘要。
- [x] 实现 `MemoryStore` 的检索与写回入口。
- [x] 定义写回策略、冲突策略和污染风险标记。
- [x] 明确哪些内容进入长期记忆，哪些仅进入场次摘要。

阶段 4 已落地范围：

- 本地 Tauri 命令 `load_memory_snapshot`、`retrieve_memory`、`submit_memory_write`。
- `MemoryStore` 使用本地 store 持久化，首次启动写入默认种子快照。
- 观众信息页和直播回顾页读取真实记忆快照，不再依赖 `mockSnapshot` 的业务样例。
- 写回入口为 API 优先，不新增页面按钮或调试表单；workbench 在互动投递成功后提交对应候选写回，并回显接受、拒绝、隔离或提交失败结果。
- 场次摘要承接短期事实，长期互动事实只接受重复出现或主播确认的稳定内容；冲突、低置信和污染风险进入拒绝或待确认记录。

完成判定：

- 生成前可检索到稳定的结构化记忆片段。
- 生成后能按规则写回，不会无条件污染长期数据。

## 阶段 5：模拟编排

目标：让系统先决定“是否发、谁来发、发什么类型”，再调用模型生成内容。

- [x] 定义观众人设池和基础同接模型。
- [x] 建立冷场、平稳、高潮、收束等节奏状态。
- [x] 实现互动频率、概率、间隔和并发控制。
- [x] 生成待执行互动意图，交给生成层完成具体内容。

阶段 5 已落地范围：

- 新增本地 `AudiencePlanner`，按上下文窗口、记忆观众、影子观众池和队列压力生成互动意图。
- 观众池分为 `memory_profile` 与 `shadow_profile`：记忆观众来自 `MemoryStore`，影子观众仅作为本地低成本群体画像，不默认写入长期记忆。
- 编排层使用确定性抽样、观众冷却、高价值互动预算和通道间隔限制，避免连续刷屏和不合理消费。
- 本地 mock 数据源改为 planner 驱动，每个 tick 先提交上下文，再按互动意图产出本地事件并进入同一 blivechat 队列。
- mock 数据源只把成功投递的 `memory_profile` 候选交给记忆写回，影子观众和被节流互动不写入长期记忆。
- 批量生成接口已固定为压缩上下文、激活观众短画像和互动意图数组；当前本地 mock 源优先批量调用 provider，失败时使用本地文案兜底，不逐观众调用 provider。
- 首页显示节奏状态、激活观众、批量生成次数、本地生成数量和冷却/节流/预算拒绝计数。

完成判定：

- 系统能在不同直播状态下输出不同密度与强度的互动意图。
- 不会出现连续刷屏、无节制高价值消费或明显失真的行为模式。

## 阶段 6：前端互动呈现

目标：把结构化互动稳定投递到改造后的 blivechat 前端。

- [x] 定义最小 `EventEmitter` 与前端事件队列。
- [x] 改造弹幕姬页以接收本地 blivechat 模拟事件。
- [x] 用本地队列记录 `enqueue`、`deliver`、`throttle`，并驱动互动时间线与投递统计。
- [x] 用同一事件模型覆盖弹幕、礼物、SC、舰长 4 类本地模拟互动。
- [x] 提供手动启停、单步、重置的本地 mock 数据源，循环触发上下文测试输入与 4 类互动输出。
- [x] 首页按 `danmaku`、`gift`、`super_chat`、`membership` 分别渲染本地 blivechat 通道。
- [x] 本地节流原因、mock source 错误和上下文错误已进入可见 UI。
- [ ] 接入后续真实平台或外部 blivechat 事件源时，补齐平台侧失败和丢弃原因。

阶段 6 已落地范围：

- 前端本地 `BlivechatEventQueue` 维护 pending 队列和协议记录。
- 弹幕姬页面不再读取静态 `queueStats` 与 `recentEvents`，改由队列快照驱动。
- 首页互动输出已从通用列表升级为 4 类本地 blivechat 渲染通道，节流原因从队列协议记录派生。
- 本地 mock 数据源只用于单机测试循环，不扩展到多直播间、云调度或真实平台接入。
- 默认只做本地模拟投递，不接真实平台。

完成判定：

- 4 类互动都能以统一事件模型进入前端并被正确渲染。
- 高频输出时渲染顺序稳定，错误可追踪。

## 阶段 7：验证与观测

目标：让系统可调试、可回放、可解释。

- [x] 提供输入源状态、provider 状态、节奏状态和投递状态面板。
- [x] 支持查看最近输入、提示组装结果、生成结果和投递结果。
- [x] 支持查看记忆检索与写回记录。
- [x] 支持本地 mock 数据源最近触发记录，作为最小可复现投递日志。

阶段 7 已落地范围：

- 首页保留轻量工作台主视图，运行观测详情通过异步组件按需加载。
- `WorkbenchMockDataSource` 在 planner 生成后发出 plan trace，store 将其解耦映射为 prompt 组装、生成结果、记忆检索、写回候选和投递协议记录。
- 当前生成结果明确标记为 Provider 生成、本地兜底或 Provider 失败后的本地兜底，不把本地文案伪装成真实模型生成。
- 记忆观测区展示检索快照和写回候选；mock 数据源不自动提交长期记忆写回，避免污染稳定数据。

完成判定：

- 能回答“为什么发了这条互动”和“为什么没发”。
- 主要失败路径都有可见日志或状态提示。

## 当前不纳入 V1

- 多直播间并发管理。
- 云端编排与多操作者协同。
- 真实平台后台、账号体系和权限系统。
- 房管、抽奖、活动脚本、复杂运营编排。
- 重度插件系统和非 OpenAI 兼容协议支持。
