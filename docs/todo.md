# BigV 开发 TODO

本文档只记录 V1 主闭环的可执行路线，不记录远期愿景。

## 阶段 1：基础重命名与壳层改造

目标：把当前初始桌面壳转换为 BigV 的产品语义和导航骨架。

- 更新应用名、产品标题、图标、配置前缀和文案。
- 重命名首页、设置页、扩展页等初始占位语义。
- 规划 BigV 的基础导航：弹幕姬、额度检查、观众信息、直播回顾、设置。
- 保留现有 Tauri/Vue 壳结构，不提前引入业务逻辑耦合。

完成判定：

- 桌面壳与文档不再出现初始工程命名。
- 基础导航能映射到 BigV 的真实页面职责，并不再保留模板占位页面。

## 阶段 2：Provider 与配置

目标：建立 OpenAI 兼容 provider 的最小可用接入。

- 定义 provider 配置项：Base URL、API Key、模型名、采样参数、超时。
- 实现连通性测试和错误回显。
- 固定 `ProviderClient` 的统一输入输出与错误模型。
- 明确结构化输出约束的承载方式。
- Provider 配置由本地 Rust store 托管，阶段 2 连通性测试固定走 `chat/completions + jsonObject`。

完成判定：

- 能从本地配置发起一次 OpenAI 兼容请求并看到明确结果或错误。
- 上层模块无需关心具体厂商差异。

## 阶段 3：输入接入

目标：打通主播语音主链路，并预留增强输入入口。

- [x] 接入主播语音文本输入，形成统一上下文事件。
- [x] 预留 Echo-Live 输入适配器接口。
- [x] 预留视觉摘要输入接口，不直接处理原始视频流。
- [x] 建立上下文时间窗和最近事件聚合逻辑。

阶段 3 已落地范围：

- 本地 Tauri 命令 `submit_context_event`、`load_context_window`、`clear_context_window`。
- 弹幕姬页面的主播语音文本手动接入口。
- 最近 300 秒、最多 80 条的运行态上下文窗口。
- 输入源状态展示：主播语音真实随窗口变化，Echo-Live 与视觉上下文保持预留接口状态。

完成判定：

- 系统能稳定消费主播语音文本并更新当前上下文窗口。
- Echo-Live 与视觉输入可按同一事件模型挂接。

## 阶段 4：记忆系统

目标：建立结构化长期记忆和场次摘要能力。

- 定义 4 类记忆的存储边界：主播设定、长期互动事实、观众画像、场次摘要。
- 实现 `MemoryStore` 的检索与写回入口。
- 定义写回策略、冲突策略和污染风险标记。
- 明确哪些内容进入长期记忆，哪些仅进入场次摘要。

完成判定：

- 生成前可检索到稳定的结构化记忆片段。
- 生成后能按规则写回，不会无条件污染长期数据。

## 阶段 5：模拟编排

目标：让系统先决定“是否发、谁来发、发什么类型”，再调用模型生成内容。

- 定义观众人设池和基础同接模型。
- 建立冷场、平稳、高潮、收束等节奏状态。
- 实现互动频率、概率、间隔和并发控制。
- 生成待执行互动意图，交给生成层完成具体内容。

完成判定：

- 系统能在不同直播状态下输出不同密度与强度的互动意图。
- 不会出现连续刷屏、无节制高价值消费或明显失真的行为模式。

## 阶段 6：前端互动呈现

目标：把结构化互动稳定投递到改造后的 blivechat 前端。

- 定义 `EventEmitter` 与前端事件队列。
- 改造 blivechat 前端以接收本地模拟事件。
- 分别实现弹幕、礼物、SC、舰长 4 类渲染通道。
- 记录投递成功、失败、节流和丢弃原因。

完成判定：

- 4 类互动都能以统一事件模型进入前端并被正确渲染。
- 高频输出时渲染顺序稳定，错误可追踪。

## 阶段 7：验证与观测

目标：让系统可调试、可回放、可解释。

- 提供输入源状态、provider 状态、节奏状态和投递状态面板。
- 支持查看最近输入、提示组装结果、生成结果和投递结果。
- 支持查看记忆检索与写回记录。
- 支持基本事件回放或最小可复现日志。

完成判定：

- 能回答“为什么发了这条互动”和“为什么没发”。
- 主要失败路径都有可见日志或状态提示。

## 当前不纳入 V1

- 多直播间并发管理。
- 云端编排与多操作者协同。
- 真实平台后台、账号体系和权限系统。
- 房管、抽奖、活动脚本、复杂运营编排。
- 重度插件系统和非 OpenAI 兼容协议支持。
