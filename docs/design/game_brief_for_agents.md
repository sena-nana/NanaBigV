# GameMaoMao 游戏设计简报（给 Agent）

这份文档用于让新的编码 Agent 快速理解当前讨论结论、项目方向和下一步实现重点。

## 1. 项目一句话定位

GameMaoMao 是一个 Godot 4 后宫资源管理 + 角色自由对话模拟器。

核心不是传统恋爱后宫，也不是单纯 Galgame 分支剧情，而是：

> 以后宫为棋盘的皇权资源管理游戏。

玩家扮演皇帝，通过分配精力、国库、皇威、情报和注意力来平衡妃嫔、外戚、皇后、宦官、宫女、太医和外朝派系。

## 2. 核心设计共识

### 后宫不是恋爱列表，而是政治地图

- 妃子 = 派系代理人。
- 皇后 = 礼法、秩序、嫡系继承权。
- 宦官 = 信息流、传话权、宫门控制。
- 宫女 = 近身信息、流言、被收买风险。
- 子嗣 = 未来继承权筹码。
- 临幸 / 召见 = 皇权注意力和继承机会的分配。

### 每个选择都应有副作用

不要设计纯收益按钮。理想效果是：

- 不召见某妃：节省精力，但本人怨恨和背后派系不满上升。
- 召见某妃：提升恩宠和派系满意，但其他妃嫔妒意上升。
- 让某妃有孕：该派系获得继承筹码，同时成为其他派系攻击目标。
- 赏赐皇后：稳定礼法秩序，但宠妃可能认为皇帝疏远自己。
- 调查贤妃：获得情报，但目标警觉和怨恨上升。

核心体验应是：

> 我知道每个选择都会制造问题，但我必须选择。

## 3. 最新交互结论：全局不要主打自由输入

此前原型采用“自由输入圣旨”作为全局入口。讨论后修正为：

> 全局玩法结构化，人物互动自由输入。

### 为什么全局自由输入不是最佳主入口

全局自由输入容易造成“空白页压力”：

- 玩家不知道当前能做什么。
- 玩家不知道行动成本和系统边界。
- 玩家会担心 AI 误解命令。
- 资源管理的可比较性会变弱。

策略层更需要清楚的行动、成本、风险和预览。

### 推荐最终结构

```text
全局阶段：事务卡 / 行动按钮 / 资源预览 / 明确目标选择
人物阶段：Galgame 对话 / 自由输入话术 / AI 解析语气与意图
规则阶段：RuleResolver 负责真实结算
叙事阶段：AI 或模板负责生成角色反应与宫中回报
```

换句话说：

```text
策略选择：结构化
话术表达：自由输入
世界结算：规则化
角色反应：AI 辅助
```

## 4. 全局玩法应该怎么呈现

全局层不要只给输入框。建议改成“奏折 / 事务卡 / 后宫事件”形式。

示例：

```text
【贵妃称病】
萧贵妃三日未出宫门，将门满意度下降。

可选：
[召见贵妃]
[赏赐萧家]
[命皇后探视]
[暗查贵妃宫]
[暂不处理]
```

每个行动要有成本和风险预览：

```text
召见贵妃

消耗：
- 精力 -28

可能影响：
- 贵妃恩宠上升
- 将门满意度上升
- 皇后猜疑上升
- 其他妃嫔妒意上升
- 可能触发怀孕检查
```

全局行动建议包括：

- 召见
- 赏赐
- 调查
- 安抚
- 太医调理
- 压下流言
- 举办宴会
- 调整宫人
- 册封 / 贬斥
- 处理奏折
- 休息

## 5. 自由输入应该放在哪里

自由输入适合放在具体人物互动中，尤其是：

- 召见妃子时说什么。
- 安抚皇后时如何解释。
- 审问宫女时如何施压。
- 试探宦官时如何套话。
- 面对孕妃时如何承诺。
- 处理妃嫔争执时如何裁断。

示例流程：

```text
玩家选择：召见贵妃
进入对话：
萧贵妃：陛下今日来，是想见臣妾，还是想稳住萧家？

玩家自由输入：
朕确实要稳住萧家，但也是真的想见你。只是你兄长最近锋芒太露，朕不能让文官抓住把柄。
```

AI / 解析器将其转成对话意图，而不是全局行动：

```json
{
  "tone": "坦诚但警告",
  "intent": "安抚贵妃并要求将门收敛",
  "emotional_message": "仍然重视她",
  "political_message": "萧家需要降调"
}
```

规则层再结算：

```text
贵妃怨恨 -6
贵妃信任 +3
军方疑虑 +2
文官警惕 -2
```

## 6. 可保留“补充口谕”，但不要作为主入口

全局行动可以有可选自由文本，但必须先确定结构化行动。

示例：

```text
行动：赏赐
目标：皇后
金额：百两黄金
公开程度：公开

补充口谕：
告诉皇后，朕仍要她替朕稳住六宫，但不要借机苛责贵妃。
```

解析后：

```json
{
  "base_action": {
    "type": "gift",
    "target": "consort_queen",
    "money_cost": 100,
    "visibility": "public"
  },
  "free_text_modifier": {
    "tone": "respectful_but_warning",
    "message": "maintain_order_without_attacking_guifei"
  }
}
```

## 7. 当前已搭建项目框架

当前仓库已经有一个 Godot 4 原型框架，但它仍偏向“全局自由输入圣旨”技术验证。

现有主要模块：

```text
project.godot
scenes/main/Main.tscn
src/ui/Main.gd
src/core/GameState.gd
src/core/RuleResolver.gd
src/core/FactionSystem.gd
src/core/PregnancySystem.gd
src/core/GoalPlanner.gd
src/core/ActionSchema.gd
src/ai/ActionParser.gd
src/ai/AIClient.gd
src/ai/PromptBuilder.gd
src/ai/ResponseValidator.gd
data/characters.json
data/factions.json
data/action_definitions.json
```

当前 `Main.gd` 的输入框和 `ActionParser.gd` 仍可作为技术原型保留，但后续应逐步重构为：

```text
GlobalCourtView：结构化事务卡和行动选择
ActionBuilder：由按钮 / 选项构造结构化 action
ConversationView：具体人物对话场景
DialogueIntentParser：解析玩家自由输入话术
RuleResolver：仍是唯一世界状态裁决者
```

## 8. 规则层必须保持权威

AI 可以负责：

- 解析玩家话术。
- 生成角色语气。
- 生成宫中回报。
- 总结记忆。
- 制造不同 NPC 对同一事件的主观说法。

AI 不应该直接决定：

- 怀孕是否成功。
- 子嗣性别和健康。
- 阴谋是否成功。
- 派系数值变化。
- 财政收入支出。
- 精力消耗。
- NPC 死亡。
- 结局判定。

这些必须由规则系统或受控随机系统决定。

原则：

> AI 负责像人一样表达，规则系统负责像游戏一样结算。

## 9. 核心数值与角色属性

### 全局资源

- `energy`：精力，限制召见、调查、处理事务。
- `treasury`：国库 / 财力，限制赏赐、太医、宴会、安抚。
- `authority`：皇威，限制压制流言、惩罚、强制裁断。
- `intel`：情报，影响调查质量和误判风险。
- `palace_tension`：后宫紧张度，影响流言、嫉妒和阴谋概率。

### 妃嫔指标

- `affection`：恩宠。
- `loyalty`：忠诚。
- `resentment`：怨恨。
- `jealousy`：妒意。
- `health`：健康。
- `influence`：影响力。
- `pregnancy_progress`：有孕迹象 / 怀孕相关状态。

### 派系指标

- `power`：权势。
- `satisfaction`：满意度。
- `suspicion`：猜疑。
- `jealousy`：派系间嫉妒或竞争压力。

## 10. 角色与派系方向

初期建议控制角色数量，不要扩太快。

当前 demo 角色方向：

- 沈皇后：礼法、正统、秩序、嫡系继承权。
- 萧贵妃：军方、将门、强势、善妒、子嗣诉求强。
- 陆淑妃：文官清流、聪慧、温和但有政治判断。
- 许贤妃：商税财权、圆滑、擅长用钱买安全。
- 林兰嫔：弱势、健康差、缺少外援、容易成为宫斗靶子。
- 王公公：宦官信息网络、谨慎逐利、可能坐大。

## 11. 插件策略

当前阶段不要堆插件。

### 第一阶段

- Godot 4.x。
- 内置 Control UI。
- GDScript 规则系统。
- JSON 数据。
- 不强依赖 AI API。
- 不强依赖 Dialogic。

### 第二阶段

可以引入 Dialogic 2，但只作为视觉小说表现层：

- 立绘。
- 表情。
- 对话框。
- 转场。
- 已编排剧情片段。

不要把核心规则、怀孕、派系、资源结算放进 Dialogic timeline。

### 更后期

- 如果要做后宫地图和可见 NPC 行动，再考虑 LimboAI。
- 如果长期记忆、事件日志、关系网络过大，再考虑 SQLite。

## 12. 内容边界

题材涉及后宫和临幸，但实现时应保持抽象、策略化、政治化。

推荐表达：

- 召见
- 承恩
- 留宿
- 子嗣
- 太医请脉
- 宫中流言
- 派系反应

避免把玩法做成露骨色情内容。核心是皇权、继承、派系和资源分配。

## 13. 下一步推荐开发任务

优先级从高到低：

1. 将全局输入框重构为“事务卡 + 行动按钮 + 成本预览”。
2. 新增 `ActionBuilder.gd`，由 UI 选项构造结构化 action。
3. 保留 `ActionParser.gd`，但将其定位改为“补充口谕 / 对话话术解析”。
4. 新增 `ConversationView.tscn`，用于单个角色 Galgame 对话。
5. 新增 `DialogueIntentParser.gd`，输出 tone / intent / promise / threat / probe 等对话意图。
6. 让 `RuleResolver` 支持“base_action + dialogue_modifier”的结算。
7. 将事件显示改成事务卡：贵妃称病、兰嫔体弱、贤妃银钱异常、皇后请整肃后宫等。
8. 加入角色短期记忆，使对话反馈能引用最近几回合事件。

## 14. 开发时不要做的事

- 不要把全局玩法继续扩成纯自由输入游戏。
- 不要让 AI 直接改 `GameState`。
- 不要把所有逻辑写进 UI 脚本。
- 不要过早引入大量插件。
- 不要快速扩充几十个妃子，先把 5 个核心角色做深。
- 不要只显示数值变化，要给玩家因果说明。
- 不要只给叙事文本，也要显示成本、风险和结算结果。

## 15. 当前一句话总结

GameMaoMao 的目标不是“AI 随便编故事的后宫聊天游戏”，而是：

> 一个结构化后宫资源管理游戏，在具体人物互动中使用 AI / 自由输入增强话术、人格和叙事反馈。
